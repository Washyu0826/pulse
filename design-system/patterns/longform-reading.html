一、為什麼是排程而不是對話

這批收藏裡最一致的訊號：把 Claude Code 當「會寫程式的 cron job」而非聊天對象。三位作者不約而同把日常雜事（報帳、週報、log 巡檢）改寫成可重跑的指令檔，再交給排程器執行12。對話式用法的成本在於每次都要重新給上下文；排程式用法把上下文固化在 CLAUDE.md 與指令檔裡，邊際成本趨近於零。

收藏時你寫下的理由「報帳那篇的 prompt 結構可以直接搬到我的電子報管線」——材料包生成時把這句餵進了蒸餾提示。

二、實測中反覆出現的三個坑

把七篇的失敗經驗對齊後，重複出現的只有三件事：

  • 權限邊界：無人值守執行時，一個未預期的確認提示就會卡住整條管線——先在沙箱列白名單，不要直接 --yolo3。
  • 長任務的中途漂移：超過 30 分鐘的批次工作要切檢查點，讓每一步可重入2。
  • 繁中輸出的格式約束：表格與全形標點要寫進系統提示，否則跑十次會有一次走樣4。

三、可以直接做的下一步

最小可行版本是一條三步管線：早上 7 點排程喚起 → 讀固定資料夾的新檔案 → 產出摘要寄到信箱。PTT 那篇的作者用 Windows 工作排程器加一個 12 行的批次檔就完成了4，與你現有的每日電子報架構幾乎同構，移植成本最低。
