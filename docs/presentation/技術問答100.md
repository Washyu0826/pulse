## A. 系統架構與設計決策

**A1. Pulse 整體架構是什麼？**
Monorepo，五個部分：`api`（FastAPI 業務 API）、`web`（Next.js 前端）、`workers`（爬蟲＋Airflow DAG＋pipeline 編排）、`ml`（情緒/事件/品質的純運算）、`docs`（ADR＋進度＋問題紀錄）。資料流是「多源爬蟲 → DQC 品質過濾/去重 → 情緒分析 → 事件偵測 → API → 前端」，用 Airflow 排程串起來。

**A2. 為什麼用 monorepo 不拆多 repo？**
個人專案、共用 model/型別頻繁（爬蟲輸出的 dict 形狀、Post schema 多處共用），monorepo 改一處能一起測、一起 CI。代價是邊界要自律（用套件分層：crawlers 只輸出 dict、services 才碰 DB）。規模再大才考慮拆。

**A3. 為什麼 ml 跟 api 分開？**
`ml` 是純函式（情緒/事件/品質），無 DB、無網路、好單元測試、可在不同直譯器跑（GPU 腳本、Airflow venv）。`api` 負責 DB I/O。這個分層讓「邏輯」和「副作用」分離，也讓 ml 不被 torch 以外的依賴綁住。

**A4. 資料流為什麼這樣排（爬→品質→情緒→事件）？**
品質要先過濾，否則 bot/垃圾/重複會污染情緒和事件（ADR-009）。情緒在事件前，因為「口碑翻轉」事件需要情緒分數。事件最後，因為它要彙整所有訊號。

**A5. 跟 HackerNews 的本質差異？**
HN 是「單一站、貼文串、只有讚數」；Pulse 是「跨來源、以模型為維度彙整、有情緒口碑、有自動事件偵測、有去重、有決策建議」。HN 是原始資料，Pulse 是把原始討論轉成決策訊號。

**A6. 為什麼選這六個模型？**
GPT/Claude/Gemini/Grok/Llama/DeepSeek 是 2026 主流、討論量足、且涵蓋閉源（GPT/Claude/Gemini/Grok）與開源（Llama/DeepSeek）兩類，能對比。用關鍵字別名比對（單一來源定義在 `keywords.py`）。

**A7. 系統可擴展性如何？加第七個模型要改幾處？**
模型定義集中在 `keywords.MODEL_KEYWORDS` ＋ `models` 表 seed（`seed_models.py`）。加模型主要改這兩處；爬蟲/情緒/事件都是泛用的，不用動。ml 的 DQC 有重列一份別名（已註明要同步，是已知的小技術債）。

**A8. 如果要做即時（real-time）而不是 15 分鐘批次？**
目前是 micro-batch（Airflow 15 分鐘）。要更即時得改成 streaming（如 Kafka + 消費者），但對「輿情趨勢」而言 15 分鐘已足夠，且批次更省、更好除錯。這是刻意的取捨：價值密度不需要秒級。

**A9. 整個系統最大的單點失敗在哪？**
業務 PostgreSQL。所有狀態都在它。緩解：Docker volume 持久化、健康檢查、Airflow 等 DB healthy 才起。生產要加備援/備份。爬蟲/情緒是無狀態、可重跑，不是單點。

**A10. 你怎麼確保改一個地方不會弄壞別的？**
~150+ 自動化測試（純函式單元 + DB 整合）＋ CI 綠燈門檻 ＋ 每階段獨立 code review。冪等設計讓重跑安全。ADR 記決策、issues-log 記坑，避免重蹈。

## B. 資料來源與爬蟲

**B1. 為什麼不用 Reddit（原本規劃）？**
拿不到 API key。與其卡住，改採「多個免 key 來源」策略，反而分散風險、零成本。架構上爬蟲是插拔式的，換來源不影響下游。

**B2. 目前有哪些來源？各自怎麼拿？**
免 key：HackerNews（Algolia API）、Dev.to（官方 API）、Lobsters（官方 API）、Hugging Face（API）、GitHub Releases（API，可選 token 提額度）。選配（需登入/cookie，best-effort）：Threads（Selenium）、X/Twitter（twscrape + cookie）。

**B3. 爬蟲的職責邊界是什麼？**
只做「抓 raw ＋ 輕量關鍵字標記」，輸出統一形狀的 dict。品質過濾、去重、情緒都交給下游（ADR-009 職責分離）。所以爬蟲是純粹、好測、好換的。

**B4. 爬蟲怎麼測？不會打到真網路嗎？**
把「正規化」抽成純函式（如 `normalize_hn_hit`、`normalize_tweet`），用固定 dict 測欄位對應、截斷、缺值處理，完全不碰網路。實際抓取的產生器只在手動/整合時跑。

**B5. 多來源同一篇文章怎麼處理？**
DQC 的跨來源去重：URL 正規化（去 utm 等追蹤參數、www、fragment）完全相同就合併；沒 URL 的用標題 SimHash + Jaccard。留最早發佈那篇當 canonical，其餘標 `DUPLICATE`，下游只算 canonical。

**B6. HackerNews 用 Algolia 而不是官方 firebase API，為什麼？**
Algolia 的 `search_by_date` 支援關鍵字搜尋＋時間排序＋日期範圍（numericFilters），適合「依關鍵字增量抓」。官方 firebase 要自己抓全站再過濾，太重。

**B7. 增量爬取怎麼做？會重複抓嗎？**
每關鍵字抓最新一頁（`search_by_date`）；重複用 `(source, external_id)` 唯一鍵 UPSERT 處理——重抓同一篇只更新讚數/留言數，不會產生重複列。

**B8. 回填歷史資料怎麼做？**
Algolia `numericFilters(created_at_i)` 做日期範圍 + 分頁。注意每查詢上限約 1000 筆，超過要把區間切更窄（逐週）分多次。時間戳要 tz-safe（naive 視為 UTC，否則差 8 小時）。

**B9. X/Twitter 為什麼這麼難爬？你怎麼解？**
2026 X 無免費官方 API、Nitter 多半失效、snscrape 失效、search 要登入。唯一可行是非官方 client（twscrape）+ 帳號 cookie。定位為選配 best-effort，預設關閉、無 cookie 即 no-op，建議用次要帳號（違反 ToS、低量風險小但非零）。

**B10. 爬蟲失敗了會怎樣？會整批掛掉嗎？**
不會。單一關鍵字/單筆解析失敗只記 log、跳過，不中斷整批（優雅降級）。網路層用 tenacity 對逾時/5xx/429 做退避重試。

## C. 資料品質 DQC

**C1. quality_score 怎麼算？**
扣分制：`score = clamp(100 + Σ 扣分, 0, 100)`，從 100 起跳，每個品質問題扣分。`DELETED` 直接歸零。這樣可解釋——每一分都對應一個明確的 flag。

**C2. 有哪些 flag？怎麼分類？**
結構（TOO_SHORT/LINK_HEAVY）、垃圾版面（EMOJI_SPAM/ALL_CAPS/EXCESSIVE_PUNCT/LIKELY_BOT）、推廣（SPAM_PHRASE/AD/SEO/AFFILIATE/CLICKBAIT/JOB_POSTING）、相關性（KEYWORD_NOT_IN_BODY/WEAK_KEYWORD）、可靠性（SARCASM_DETECTED，只標記不扣分）。

**C3. 為什麼用規則（heuristic）不用 ML 分類器？**
可解釋、快、不用標註資料、適合 24/7 排程。品質分數要能對使用者交代「為什麼這篇低分」。ML 分類器是黑盒、要訓練資料。這是刻意取捨：透明度 > 邊際準確率。也符合「需要 LLM 才用，且優先地端」的原則。

**C4. 相關性怎麼判斷「真的在談這個模型」vs「只是提到」？**
去掉 URL/code 後看關鍵字在不在「標題或正文」：只在 URL/code 裡 → `KEYWORD_NOT_IN_BODY`；正文只出現一次又不在標題 → `WEAK_KEYWORD`（提到但非主題）。

**C5. 你說 SEO 一開始誤判 600 多篇，根因？**
HN 內文是 HTML escape 過的（`I&#x27;ve`），我的 hashtag regex `#\w+` 把 `&#x27;` 裡的 `#x27` 當成 hashtag，≥3 個就判 SEO。解法：先 `html.unescape` + 去標籤；hashtag regex 改成 `(?<![\w&])#\w+`（# 前不能是 `&`）。

**C6. SEO 的關鍵字堆砌怎麼量？為什麼不用絕對次數？**
絕對次數對長文必然超標（長文自然重複詞）。改用**密度**：某 token 佔非停用詞 token > 8% 且 ≥5 次，且**排除模型別名**（談某模型本就會重複其名）。

**C7. 去重的演算法細節？為什麼不是 O(n²)？**
兩階段分桶避免全配對：(1) URL 正規化完全相同 → 直接同桶（O(n)）；(2) 標題 64-bit SimHash，切成 4×16-bit 做 banding，只有同 band 的才比 Hamming 距離 + token Jaccard。用 union-find 合併兩種證據成 cluster。

**C8. SimHash 的 banding 為什麼能保證抓到近似的？**
鴿籠原理：64 bit 切成 4 段各 16 bit，若兩個指紋 Hamming 距離 ≤3，最多 3 個 bit 不同，必有至少一段完全相同 → 一定落在同一個 band，不會漏配。再用 Jaccard ≥0.8 做最終確認（雙門檻保守、低誤判）。

**C9. cluster 裡留哪一篇？怎麼決定 canonical？**
依序：最早 `posted_at`（原始那篇）→ 互動高（讚+留言）→ 來源優先序 → 最小 id。用最早發佈當主，所以重抓互動數變動也不會讓 canonical 跳來跳去（穩定、idempotent）。

**C10. 去重結果怎麼存？為什麼不開新表？**
存進現有 `quality_flags`（text[]）：非 canonical 標 `DUPLICATE` + `CANONICAL:<id>`。不開新表是最小侵入、零 migration。代價是 text[] 沒有 FK 完整性——讀的人要容忍 dangling id。要關聯查詢時再升級成 `canonical_id` 欄位。

## D. 情緒分析（ML）

**D1. 用什麼模型？為什麼？**
`cardiffnlp/twitter-roberta-base-sentiment-latest`（RoBERTa）。它在**社群短文**上訓練（推文），跟我們的資料分佈（社群討論）接近，三分類（正/中/負）。比通用情緒模型或關鍵字法準。

**D2. 跑在哪？效能如何？**
本機 GPU（RTX 4060），batch 推論。5000 多篇一次批次跑完。刻意**不放進 Airflow image**（torch ~2GB、容器內只能 CPU、慢），維持 Airflow 輕量；情緒走獨立 GPU 腳本，是更好的資源隔離。

**D3. 模型輸出的機率可以直接信嗎？**
不行，神經網路常 over-confident。我做 **temperature scaling**（溫度校準，Guo 2017）把 logits 除以溫度再 softmax，讓信心更貼近真實正確率。

**D4. 低信心的怎麼處理？**
**信心棄答帶**（abstention，Xin 2021）：信心低於門檻的標記為低信心，聚合時降權或排除，避免模型在難例上瞎猜污染口碑。

**D5. 口碑指數怎麼算？不是平均嗎？**
不是簡單平均。是**信心加權的 soft 聚合**：用每篇的 `score*(p_pos - p_neg)` 加權求和除以信心總和（Dawid-Skene 式概念），再做**小樣本收縮** `n/(n+3)`，最後 ×100。所以少量樣本不會給極端值。

**D6. 為什麼要小樣本收縮？背後是什麼？**
Beta-Binomial / 收縮估計的概念：樣本少時往中性拉（Brown 2001 的 interval 思想）。3 篇就 +100 不可信；收縮讓 n 小時自動保守，n 大時逼近真實。

**D7. 「口碑翻轉」怎麼判定不是雜訊？**
**兩比例 z 檢定**：比較近 14 天 vs 前 14 天的正面比例，算 z 值和 p-value，只有統計顯著（如 p<0.05）才報翻轉。所以不是「數字變了就報」，是「變化在統計上站得住腳才報」。

**D8. 分歧度（polarization）是什麼？**
衡量「大家意見是否分裂」（Morales 2015）。即使平均口碑中性，也可能是「一半超愛一半超恨」——分歧度能抓到這種兩極化，比單一平均更有資訊。

**D9. 反諷（sarcasm）怎麼辦？情緒模型不是會錯嗎？**
會。所以用 regex 標記反諷訊號（`/s`、「oh great another」等），標起來給下游/人工參考（只 flag 不直接改分），因為反諷本來就是情緒模型的已知弱點，誠實標示比假裝準確好。

**D10. 你說整合了 30 篇論文，舉幾個實際落地的？**
溫度校準（Guo 2017）、信心棄答（Xin 2021）、信心加權聚合（Dawid-Skene 1979）、小樣本收縮（Brown 2001）、兩比例 z 檢定判翻轉、極化分數（Morales 2015）、反諷標記（Joshi 2017）。清單在 `docs/research/sentiment-literature.md`，標了哪些已實作、哪些是未來。

**D11. 情緒模型的偏誤/限制？**
訓練在英文推文，非英文、領域術語、長文會掉準（已用校準+棄答緩解）。我們**不做語言過濾**（使用者決定），所以非英文照樣評分，這是已知限制。口碑只是訊號，不是真理。

**D12. 怎麼驗證口碑算得對？**
人工抽樣對照（看單篇情緒與卡片口碑是否一致）＋ 邏輯測試（無資料→None、全正面→高分但被收縮壓低、翻轉要 z 顯著）。沒有 ground-truth 標註集是限制，目前靠合理性檢查。

## E. 事件偵測（統計）

**E1. 「討論突增」怎麼偵測？為什麼不用平均+標準差？**
用**穩健修正 z-score**（median + MAD），不用 mean/std。因為討論量是**偏態、有離群**的——一次爆量會把 mean/std 拉高，反而遮蔽偵測。median/MAD 對離群穩健。

**E2. MAD 是什麼？修正 z-score 公式？**
MAD = median(|x - median(x)|)，中位數絕對離差。修正 z-score = `0.6745 * (x - median) / MAD`（0.6745 讓 MAD 在常態下逼近標準差）。超過門檻（如 3.5）判突增。

**E3. MAD = 0 怎麼辦？（很多天沒討論）**
會除以 0。要特判：median/MAD 都接近 0（平日幾乎無討論）時，用「從幾乎無人討論突然爆量」的分支判定，而不是讓公式爆掉或遮蔽最強的突增（這是 code review 抓到的真實 bug）。

**E4. 時間序列有空洞（某天 0 篇）怎麼處理？**
`fill_daily_gaps`：把缺的日期補 0，否則 median 會被高估、突增判不出來。先補滿連續日期再算。

**E5. 突增的「主因」怎麼來的？**
偵測到某模型某天突增時，取那天**讚數最高**的貼文標題當「主要討論」，存進事件。讓使用者看到「為什麼那天爆」（仿 Brand24 的 "this spike was caused by X"）。

**E6. 「發布事件」怎麼聚合？**
HF/GitHub 的 release 依 (模型, 日) 分組聚合（同一天多個版本算一個發布事件），過濾 GitHub prerelease（-rc 等）降噪。

**E7. 事件會不會重複產生？冪等嗎？**
每個事件有 `dedup_key`（如 `discussion_spike:claude:2026-05-13`）唯一，UPSERT。重跑收斂到同狀態。

**E8. 資料回填後，舊事件不成立了怎麼辦？**
**對帳**：偵測時刪除「這次沒再偵測到」的舊事件（dedup_key 不在當前集合內）。delete + upsert 在同一交易，retry 安全。解決了「殭屍事件」問題（回填後 median 變了，舊突增不再成立卻殘留）。

**E9. 接了 DQC 品質門檻後事件數變化？**
discussion_spike 25→23：兩個由「跨來源重複 burst」灌出的假突增被濾掉了。因為突增現在只算「高品質、非重複」的貼文（quality_score≥30 且非 DUPLICATE）。這證明品質過濾真的修正了誤判。

**E10. 偵測的門檻（z 值）怎麼定？**
目前是經驗值（modified z ≥ 3.5 一帶），對照真實資料調，確認抓到的是真突增（如 Claude 5/13 的 118 篇 vs 平日中位 60）。沒有標註集，靠真實案例校準——這是限制，也是可以再嚴謹化的地方。

## F. Airflow 排程/編排

**F1. 為什麼用 Airflow 不用 cron？**
要的是「有相依、有重試、有告警、可觀測、有 backfill 控制」的編排，不只是定時。cron 沒有相依管理、失敗重試、UI 監控。Airflow 是業界資料工程標準。

**F2. 有幾個 DAG？怎麼分？**
7 個：4 個爬蟲（hackernews/community/reddit/x）、fetch_releases、event_detection、data_quality。爬蟲是 producer，事件偵測/DQC 是 consumer。

**F3. DAG 之間怎麼相依？用 sensor 嗎？**
用 **Airflow Datasets（資料感知排程）**，不用 ExternalTaskSensor。爬蟲成功標記 `POSTS` Dataset，事件偵測/DQC 訂閱它——有新資料才觸發，不空轉。比 sensor 乾淨（sensor 要對齊 execution_date、會佔 worker slot）。

**F4. 用 TaskFlow API 還是傳統 Operator？**
TaskFlow（`@dag`/`@task`）。比傳統 PythonOperator 簡潔，XCom 自動傳值，可讀性高。

**F5.（殺手題）Airflow 跟你的程式有依賴衝突，怎麼解的？**
Airflow 2.9 把 SQLAlchemy 鎖在 <2.0，但我的業務碼用 SQLAlchemy 2.0（`DeclarativeBase`/`mapped_column`/`async_sessionmaker`），無法共存於同一直譯器。解法：自建 image 內**雙環境**——主環境跑 Airflow（1.4）、獨立 venv 裝 SQLAlchemy 2.0，DAG task 用 **`ExternalPythonOperator`（`@task.external_python`）** 在那個 venv 執行業務邏輯。

**F6. ExternalPythonOperator 怎麼運作？有什麼坑？**
它把 task 函式的**原始碼**抽出來，丟到目標 venv 的直譯器跑，參數用 pickle 傳。坑：函式body 要自包含（所有 import 寫在函式內，不能用模組層的名字）；回傳值要可 pickle。truststore 也要在 venv 內注入（主環境的 plugin 到不了子行程）。

**F7. async 的爬蟲怎麼在（同步的）Airflow task 跑？**
LocalExecutor 每個 task 在獨立子行程、沒有既有 event loop，所以 `asyncio.run()` 安全。一個 task 一個 `asyncio.run`、一個 loop，把 async generator 在同一 loop 內耗盡，避免跨 loop 問題。

**F8. 重試策略？冪等怎麼保證？**
retries=3 + 指數退避（2→4→8 分，上限 30 分）。冪等靠 UPSERT（自然鍵 on-conflict）和事件的 dedup_key——重跑收斂，不會產生重複或雙重計數。

**F9. 失敗了怎麼知道？**
`on_failure_callback` 發 Slack（讀 Airflow Variable 的 webhook，best-effort，沒設就略過）。另外加了 `execution_timeout`（15 分）避免某次卡死佔住 `max_active_runs=1` 的唯一 slot 導致管線無聲停擺。

**F10. 排程怎麼避免 backfill 風暴？**
所有 DAG `catchup=False` + `max_active_runs=1` + 固定過去的 `start_date`。否則 start_date 在過去 + catchup 會一次補幾千個 run。

## G. 資料庫（PostgreSQL / SQLAlchemy）

**G1. 為什麼用 PostgreSQL？**
關聯查詢（模型↔貼文多對多、彙總）、`text[]`（quality_flags）、`JSONB`（event extra）、部分索引、window/filter 聚合都原生支援。輿情資料是結構化的，不需要 NoSQL。

**G2. 為什麼 SQLAlchemy 2.0 async？**
async（asyncpg）讓爬蟲/API 的 I/O 不阻塞。2.0 的 `Mapped`/`mapped_column` 型別友善、`DeclarativeBase` 現代。代價就是跟 Airflow 的 1.4 衝突（見 F5）。

**G3. 你踩過 PG 的什麼上限？**
單一語句 **32767 個 bind 參數**上限。回填 5000+ 列一次 INSERT 就爆。解法：抽共用 `chunked()`，所有 upsert 分塊（每塊 1000 列），單一交易末尾才 commit（保原子性）。加了靜態不變式測試（chunk×欄位 < 32767）。

**G4. UPSERT 怎麼做？衝突怎麼處理？**
`INSERT ... ON CONFLICT (source, external_id) DO UPDATE`。只更新會變的欄位（讚數/留言/標題），不動 created_at/quality_*（保留首次值與 DQC 結果）。同批次內先去重，否則 PG 報 "cannot affect row a second time"。

**G5. DQC 怎麼有效率地找「還沒處理的貼文」？**
**部分索引**：`ix_posts_dq_unprocessed ON (fetched_at) WHERE dq_processed_at IS NULL`。處理完設 `dq_processed_at` → 自動離開索引，下次查詢只掃未處理的，很便宜。

**G6. 重新評分（改了規則）怎麼做？**
把 `dq_processed_at` 設回 NULL → 重跑 DQC。評分是純覆寫、冪等，重跑安全。

**G7. Migration 怎麼管？**
Alembic autogenerate。先定 `naming_convention`（約束/索引命名規則）讓產出穩定可逆。`env.py` import models + `compare_server_default=True` 才偵測得到變更。

**G8. 那個 bulk update 為什麼不能用 ORM？**
`update(Post).where(id==bindparam).values(...)` + executemany 會走 ORM Bulk Update by PK，要求參數含真實 PK 欄名、不支援額外 WHERE 同步。改用 **Core** `update(Post.__table__)` + bindparam executemany（純 SQL `UPDATE ... WHERE id=:id`）繞過 ORM 邏輯。`text[]` 要綁 `ARRAY(Text)` 型別。

**G9. 連線埠為什麼是 5433 不是 5432？**
本機已裝 PostgreSQL 18 佔 5432，且 Windows 的 localhost 在 IPv4/IPv6 解析不定會連錯 DB。Docker host 埠改 5433、連線用 `127.0.0.1` 強制 IPv4。容器之間仍用 service 名 `db:5432`。

**G10. 跨 event loop 的測試問題？**
pytest-asyncio 每個 test function 各開新 loop，但 module-scope 的 async engine fixture 建在某個 loop → 跨 loop 用同一連線報錯。解法：engine fixture 改 **function scope**（與 test 同 loop）。

## H. 後端（FastAPI / API）

**H1. 有哪些 API endpoint？**
`/api/models`（看板）、`/api/models/{slug}`（模型詳細＋時間序列）、`/api/events`、`/api/releases`、`/api/decide`、`/api/health`，以及（監控用）`/metrics`。

**H2. 服務層怎麼分？為什麼？**
router 只管 HTTP，service 管查詢/彙總邏輯。upsert 服務自行 commit、自掌交易（DAG/腳本共用），**不**在 FastAPI 的 request session 內呼叫（會與其 commit 衝突）。

**H3. 看板的口碑/計數是即時算的嗎？效能？**
即時 SQL 聚合（count + filter + group by）。資料量（萬級）下夠快。再大可加物化視圖/快取。現在優先正確與簡單。

**H4. `/decide` 怎麼運作？用 LLM 嗎？**
**資料驅動，不依賴 LLM**：比較選定模型的口碑、討論量、議題相關熱門討論，給有證據的推薦（`_recommend`）。LLM 合成層寫好但休眠（有 key 才啟用，且優先地端）。所以無 key 也能跑、答案基於真實資料而非模型瞎掰。

**H5. 品質門檻怎麼接進 API？**
共用 `quality_post_filter()`（quality_score≥30 或 NULL 放行、排除 DUPLICATE），接進看板計數、口碑彙總、事件偵測。NULL 放行是避免新貼文在 DQC 跑之前就被丟掉。

**H6. API 怎麼測？**
純邏輯（`_recommend`、`_template_summary`）純函式單元測試；DB 相關用整合測試（function-scope engine fixture，建表→測→拆）。

**H7. 為什麼模型詳細頁要新開一個 endpoint？**
它要的是「單一模型的時間序列」（每日討論量+口碑，gap-filled）＋ 近期事件/熱門討論/發布，跟看板的「全模型摘要」資料形狀不同。新 service `model_detail.py`，複用看板的口碑公式保持一致。

**H8. CORS / 安全？**
CORS 限定前端來源（settings.cors_origins）。目前是 read-only 公開資料，無寫入 API、無敏感資料。要上線會加 rate limit、API key（監控 `/metrics` 也要保護）。

## I. 前端（Next.js）

**I1. 為什麼 Next.js 14 App Router？**
Server Components 預設在伺服器抓資料、串流到前端，首屏快、SEO 友善。App Router 的 layout/loading/Suspense 適合儀表板。

**I2. 後端掛了前端會崩嗎？**
不會。用 **fail-soft 的 ApiResult discriminated union**：每個 section 各自抓資料、各自守 ok/error，一個 API 失敗只該區塊顯示「暫無資料」，不拖垮整頁。實測後端關掉四個頁面都還 200。

**I3. 資料怎麼更新？會即時嗎？**
ISR（`revalidate: 60`）＋ per-section Suspense 串流。不是 websocket 即時，但 60 秒重新驗證對輿情足夠，且省資源。

**I4. 趨勢圖用什麼畫？為什麼不用 chart 套件？**
**純 SVG 手刻、零依賴**。因為之前 recharts 的 package-lock 造成跨平台 CI 壞掉（@emnapi 在 Linux 缺）。手刻 SVG 避免依賴風險，也夠用（面積圖+折線）。

**I5. 怎麼讓使用者「一看就懂怎麼用」？**
新 Hero（一句價值 + 與 HN 差異）、首訪「三步上手」引導條（localStorage 記憶）、術語人話化 + InfoHint（滑過才看真實指標，藏起 z-score 等黑話）、解釋性空狀態。

**I6. 搜尋/篩選怎麼做的？**
事件流可依「事件類型/模型」篩選，狀態存在 URL query——可分享、上一頁有效（用 URL 當 single source of truth，不是 local state）。

**I7. 前端怎麼跟後端串？型別怎麼對齊？**
`lib/api.ts` 統一 fetch（泛型 `fetchArray<T>`）+ `lib/types.ts` 對齊後端形狀。後端改欄位前端型別會編譯期抓到（typecheck 在 CI）。

## J. DevOps / Docker / CI / 監控

**J1. 怎麼容器化？**
Docker Compose：業務 DB、Airflow（自建 image，metadata DB 獨立）、之後加監控。原始碼 bind-mount + PYTHONPATH 注入，改 DAG/爬蟲免重 build。

**J2. CI 跑什麼？**
GitHub Actions：api job（uv sync、ruff、pytest，附 postgres service）＋ web job（npm ci、typecheck、lint、build）。每次 push/PR 到 main 跑。

**J3. 你遇到最多的坑是哪類？**
跨平台（本機 Windows vs CI Linux）＋ TLS 攔截。cp950 vs UTF-8、npm lock 跨平台、optional 依賴、憑證驗證——這些佔了 issues-log 一大半。

**J4. TLS 攔截是什麼？怎麼解？**
所在網路重簽憑證，根 CA 在 OS 信任庫但不在各語言工具的信任庫 → pip/npm/httpx/Docker build 全部 `CERTIFICATE_VERIFY_FAILED`。解法：各自指向 OS 信任庫（`--trusted-host`/`truststore`/`--use-system-ca`）；Docker 容器還要把攔截根 CA 併進容器自己的信任庫。

**J5. 監控怎麼設計？（規劃中）**
Airflow 用 StatsD 推指標 → statsd_exporter 轉 Prometheus 格式 → Prometheus 抓 → Grafana 畫。三組指標：DAG 成功率/時長（statsd_exporter）、資料量趨勢（FastAPI `/metrics` 用 prometheus-client 暴露業務 gauge）、容器資源（cAdvisor）。

**J6. 「資料量趨勢」為什麼用 pull 不用 push（Pushgateway）？**
FastAPI 是長駐的 scrape target，pull `/metrics` 才對。Pushgateway 是給「跑完就死的批次」用的，會有 stale metric 的坑，不適合這裡。趨勢由 Prometheus 定期抓 gauge 自然形成時間序列。

**J7. 系統資源為什麼用 cAdvisor 不用 node-exporter？**
node-exporter 讀 Linux host 的 /proc，在 Docker Desktop（Windows/WSL2）讀到的是隱藏 VM、沒用。cAdvisor 透過 Docker daemon 報每個容器的 CPU/mem，才是這裡要的。

**J8. 多 agent 開發是什麼意思？怎麼確保品質？**
每個階段「研究 agent → 實作 → 獨立 code review agent → 記錄」。前端大升級還用獨立 git worktree 背景 agent 平行做，與後端同時推進、互不干擾、完成後乾淨合併。品質靠獨立 review（很多 bug 是 review 抓到不是測試）+ CI。

## K. 取捨、限制、誠實面（mentor 最愛問）

**K1. 這個專案最大的限制是什麼？**
（1）情緒/事件門檻沒有標註 ground-truth，靠合理性檢查校準；（2）資料來源以英文社群為主、無語言過濾；（3）X/Threads 是 best-effort、易壞；（4）規模還小（萬級），有些做法（全表去重載記憶體）到十萬級要改增量。

**K2. 如果重做，你會改什麼？**
模型別名集中成單一來源（現在 ml/crawler/DB 三處）；去重做成增量而非全表；情緒加一個小型標註集做量化評估；更早把監控接上。

**K3. 為什麼不用現成的社群監聽工具（Brand24 等）？**
那些是泛用、針對品牌行銷。Pulse 專門針對 AI 模型，有模型維度、發布偵測、口碑翻轉、選型決策——這些通用工具沒有。且這是學習/作品集，自己做才學得到資料工程+ML 全鏈路。

**K4. 怎麼證明這不只是「爬蟲 + 撈資料」？**
三個層次：資料工程（多源、品質、去重、排程、可觀測）＋ 統計/ML（穩健 z-score 事件、校準+收縮的情緒口碑、z 檢定翻轉）＋ 產品（決策報告、自我解釋的前端）。每層都有非顯而易見的工程決策。

**K5. 下一步要做什麼？**
監控（Grafana）收尾、X 來源實接、下游品質門檻已接、之後做主題分類/摘要（優先地端 LLM）、帳號/個人化（訂閱收藏）、通知（Slack/Email/站內）。

**K6. 你從這個專案學到最重要的一件事？**
「魔鬼在資料的細節裡」——HTML 實體誤判 SEO、median=0 遮蔽突增、回填讓舊事件失效、依賴版本衝突……真實資料和真實環境的坑，遠比演算法本身難。把它們都記成 issues-log，是這個專案最實在的產出之一。
