# Pulse 簡報素材（給網頁版 Claude 生成投影片用）

> **使用說明（給網頁 Claude 的指示）**：請依下列素材，產生一份 **5–10 分鐘、約 12–16 頁**的中文簡報（技術名詞保留英文）。
> 章節固定為：**1 動機 · 2 規格 · 3 架構 · 4 目前進度 · 5 遇到難點**，外加封面與結尾各一頁。
> 風格：簡潔、工程感、深色或學術皆可；每頁重點 3–5 條、善用圖表/表格；數字要醒目（它們是真實實測值）。
> 定調：**架構與方法呈現為完整設計；成果標示「已實作 / 進行中」**（誠實但不自貶）。可加講者備註。

---

## 0. 一句話定位（封面用）

**Pulse —— 把社群討論轉成「AI 模型決策訊號」的即時情報儀表板。**

監測 6 個主流 AI 模型（GPT、Claude、Gemini、Grok、Llama、DeepSeek），從多個技術社群自動擷取真實使用評論，
經品質過濾 → 情緒分析 → 事件偵測，產出**口碑指數、即時事件流、選型決策報告與自動告警**。

副標：*AI 工程師的每日情報秘書* ｜ 與 HackerNews 的差異：HN 只有讚數，Pulse 給你**情緒、事件、彙整與建議**。

---

## 1. 動機（Motivation）

**問題背景**
- AI 模型迭代極快：幾乎每週都有新版本、新模型、定價調整、服務中斷。工程師選型與追蹤成本高。
- 現有資訊管道各有缺口：
  - **官方 changelog / blog**：只報喜，看不到真實使用反饋與踩雷。
  - **HackerNews / 社群**：只有「讚數 + 留言」，雜訊多、需人工逐篇看，**沒有模型維度、沒有情緒、沒有彙整**。
  - **Twitter/X、Threads**：即時但極度雜亂，難萃取有效訊號。

**真正的痛點（沒人解決的）**
把「跨社群的真實使用評論」轉成「**可決策的訊號**」——
- 這個模型最近**口碑好不好**？（不是讚數，是情緒）
- 最近有沒有**出包 / 翻車 / 重大發布**？（事件，不是逐篇翻）
- 做某個任務（如 coding agent）**該選 A 還是 B**？（有證據的建議）

**目標使用者**：需要選型 / 追蹤 AI 模型動態的工程師、技術決策者、研究者。

**與 HackerNews 的差異化（一頁對照表，很重要）**

| 面向 | HackerNews | **Pulse** |
|------|-----------|-----------|
| 資料維度 | 單一貼文串 | **以「模型」為維度彙整** |
| 情感 | 只有 upvote 數 | **情緒口碑指數（紅綠燈）** |
| 事件 | 人工瀏覽 | **自動偵測：討論突增 / 重大發布 / 口碑翻轉** |
| 來源 | 單一站 | **跨多來源彙整 + 去重** |
| 決策 | 無 | **選型決策報告（有證據）** |
| 通知 | 無 | **自動告警（Slack / 站內 / Email）** |

## 2. 規格（Spec / 功能）

**核心功能（對應內部代號）**
- **F2 模型看板**：6 模型即時彙總（討論量、近 7 天趨勢、最新發布、口碑指數）。
- **F8 多源事件偵測**：三類事件
  - `discussion_spike` 討論量突增（含「主因」最熱貼文）
  - `launch` 重大發布（HF / GitHub Releases 聚合）
  - `sentiment_flip` 口碑翻轉（統計顯著才報）
- **情緒 / 口碑**：每模型情緒口碑指數 + 分歧度（紅綠燈呈現）。
- **F3/F4 決策報告 `/decide`**：「Claude 還是 GPT 適合做 X？」→ 用真實討論數據給有證據的建議。
- **DQC 資料品質**：每篇貼文品質評分 + 標記（垃圾/廣告/SEO/相關性）+ 跨來源去重。
- **自動排程與告警**：Airflow 定時爬取 → 偵測 → 失敗 Slack 告警。

**監測對象**：GPT、Claude、Gemini、Grok、Llama、DeepSeek（關鍵字別名比對，單一來源定義）。

**技術規格 / Stack**
- 後端：**FastAPI** + **PostgreSQL 16** + **SQLAlchemy 2.0 (async, asyncpg)** + Alembic migration
- 排程：**Apache Airflow 2.9**（TaskFlow API、LocalExecutor、Datasets 資料感知排程）
- ML：**RoBERTa** 情緒模型（GPU）、F8 統計事件偵測（穩健 z-score）
- 前端：**Next.js 14**（App Router、Server Components、Suspense 串流、ISR）
- 基礎建設：**Docker Compose**、**GitHub Actions CI**、多 agent 開發（研究→實作→code review→記錄）
- 測試：~150+ 測試（API + ML + workers），CI 綠燈

## 3. 架構（Architecture）

**Monorepo 結構**
```
pulse/
├── api/        FastAPI + SQLAlchemy models/services（業務 DB 寫入路徑）
├── web/        Next.js 14 前端儀表板
├── workers/    crawlers/（多源爬蟲）+ pipeline/（編排）+ dags/（Airflow）
├── ml/         sentiment.py（情緒）/ event_detection.py（事件）/ data_quality.py（DQC）/ dedup.py
└── docs/       ADR 決策紀錄 + 進度紀錄 + 問題全紀錄
```

**資料流（核心 pipeline，建議畫成流程圖）**
```
多源爬蟲 ──► 原始貼文(posts) ──► DQC 品質過濾/去重 ──► 情緒分析(sentiments)
   │                                   │                        │
   └──► 發布事件(release_events) ───────┴──► F8 事件偵測(events) ◄┘
                                                  │
                                   API(FastAPI) ──┴──► 前端儀表板 / 決策報告 / 告警
```
排程：Airflow 每 15 分鐘爬取 → 以 **Dataset** 觸發下游偵測（有新資料才跑、不空轉）。

**資料來源（全部免 key 為主力；策略上很關鍵）**
| 來源 | 取得方式 | 角色 |
|------|---------|------|
| HackerNews | Algolia API（免 key） | 主力 |
| Dev.to | 官方 API（免 key） | 主力 |
| Lobsters | 官方 API（免 key） | 主力 |
| Hugging Face Hub | API（免 key） | 發布訊號 |
| GitHub Releases | API（免 key，可選 token） | 發布訊號 |
| Threads | Selenium（best-effort 選配） | 補充 |
| X / Twitter | twscrape + cookie（選配，進行中） | 補充 |

> **設計取捨**：原規劃用 Reddit，但拿不到 API key → 改採「**多個免 key 來源**」策略，反而更穩、更分散風險。

**6 個 Airflow DAG**
- `crawl_hackernews`、`crawl_community`(Dev.to+Lobsters)、`crawl_reddit`(預設關閉)、`fetch_releases` → producer
- `event_detection`（資料感知排程，吃 POSTS/RELEASE_EVENTS Dataset）
- `data_quality`（DQC，產出 POSTS_DQ_PASSED Dataset）
- 全部 retries=3 指數退避、`catchup=False`、`max_active_runs=1`、失敗 Slack 告警

**關鍵架構決策（ADR 紀錄）**
- 爬蟲只抓 raw、品質過濾交給獨立 DQC 層（職責分離）。
- 事件偵測用**穩健統計**（median/MAD modified z-score），對稀疏/偏態資料魯棒。
- 情緒口碑用**信心加權 + 小樣本收縮**，避免少量樣本給出極端值。

## 5. 遇到難點（Challenges）

> 挑 4–6 個最有故事性的講；每個都「症狀 → 根因 → 解法」。

**A. 資料來源受限 → 多源策略**
- 難點：拿不到 Reddit API key；X/Twitter 2026 已無免費官方 API、Nitter 多半失效、search 需登入。
- 解法：改採「**多個免 key 來源**」（HN/Dev.to/Lobsters/HF/GitHub）為主力；X/Threads 走 cookie 的 best-effort 選配。風險分散、零成本。

**B. 依賴版本衝突（最硬的工程題）**
- 難點：Apache Airflow 2.9 把 **SQLAlchemy 鎖在 <2.0**，但業務碼用 **SQLAlchemy 2.0**（`DeclarativeBase`/`async`）→ 無法共存於同一直譯器。
- 解法：自建 Airflow image **雙環境** —— 主環境跑 Airflow(1.4)，獨立 venv 裝 SQLAlchemy 2.0；DAG task 用 **`ExternalPythonOperator`** 在該 venv 執行業務邏輯。
- 收穫：這是把「依賴衝突」用隔離正解掉的典型案例（可寫進履歷的 MLOps 經驗）。

**C. 企業/校園網路 TLS 攔截**
- 難點：所在網路重簽憑證，pip / npm / httpx / Docker build 全部 `CERTIFICATE_VERIFY_FAILED`。
- 解法：各工具分別指向 OS 信任庫（`truststore` / `--trusted-host` / `--use-system-ca`）；容器內再把攔截根 CA 併入信任庫。

**D. 資料品質的「假陽性」**
- 難點：DQC 上線後 SEO 誤判 600+ 篇 —— HN 內文的 HTML 實體 `&#x27;` 被 hashtag 規則當成 `#x27`；又如「OpenAI is hiring」被誤判徵才、「5 reasons Claude…」被誤判 clickbait。
- 解法：先 HTML 解碼 + 去標籤；SEO 改用**密度**且排除模型別名；收緊啟發式（需多重訊號才判垃圾）。低分貼文從 32 → 3，誤判大幅下降。

**E. PostgreSQL 批次寫入上限**
- 難點：一次 UPSERT 5000+ 列觸到 PG **32767 bind 參數上限**。
- 解法：抽共用 `chunked()` 分塊，所有寫入服務統一分塊、單一交易末尾才 commit（保原子性）。

**F. 資料形狀會變 → 偵測要「對帳」**
- 難點：歷史回填讓時間序列由稀疏變密集，舊的突增事件不再成立卻殘留（殭屍事件）。
- 解法：偵測時做**對帳**（刪除這次不再偵測到的舊事件）+ 穩健 z-score（median/MAD）。

**G. 跨平台 / 本機 ≠ CI**
- 難點：Windows(cp950) vs Linux(UTF-8)、npm lock、optional 依賴 —— 跨平台差異是最常踩雷處。
- 解法：編碼強制 UTF-8、lockfile 跨平台重產、CI 鎖環境；獨立 code review 抓邏輯 bug。
